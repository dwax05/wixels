# Eww-Style Quotes Widget Proof of Concept Implementation Plan

> **Superseded during implementation (2026-07-17).** The original data-only
> manifest/bundled-helper design was intentionally replaced after the product decision
> to support trusted user commands. The implemented direction is
> `~/.config/wixels/widgets.toml`: `[[variable]]` entries run `/bin/sh -lc` on a timer,
> and `[[widget]]` text interpolates their values as `{name}`. `script-quotes` and its
> manifest/broker are removed; a random-quote shell script is now ordinary user config.
> The current implementation lives in `Sources/wixels/WidgetsConfig.swift` and
> `Sources/wixels/CommandTextWidget.swift`.

## Overview

Introduce the first non-dylib Wixels widget: a read-only Quotes card described by
a JSON manifest and sampled through a Wixels-owned helper script. The feature proves
the Eww-inspired split between declarative widget definition, a host-owned command
protocol, and the existing native Wixels renderer without accepting arbitrary shell
commands or adding a privacy-sensitive capability.

This is intentionally not an Eww integration and does not depend on Eww. “Eww-style”
means a declarative widget package plus a data command protocol: the signed Wixels
host continues to create macOS windows, schedule updates, apply themes, and enforce
which commands a manifest may request.

Proposed version: **0.3.0**. This adds a new supported extension format and therefore
expands the public extension surface beyond the current 0.2.x dylib-only model.

## Problem Statement

New widgets currently require a Swift package, a dynamic library, a matching Swift
toolchain, `dlopen` registration, and full trust in code that runs in Wixels'
process. That is too much machinery for simple data-only widgets and is not a safe
foundation for user-provided scripts. Wixels has no declarative widget format or
validated process protocol today.

## Solution Architecture

Add a second, parallel extension lane called **script widgets**:

```
wixels-script-widget.json (data only)
          |
          | validates kind, quote list, placement, and source name
          v
ScriptWidgetLoader ── registers a WixelsKit-owned WidgetSpec ──> existing WidgetHost
          |                                                   (window/theme/scheduler)
          v
ScriptWidgetBroker ── supervised stdio ──> wixels-widget-broker.pl
                                         (signed app resource)
```

The first manifest is a Quotes widget. It contains a nonempty list of literal quote
strings and uses the fixed source name `quote.random`. The host sends those values to
the bundled Perl helper in a newline-delimited JSON request; the helper returns one
validated JSON response. The helper has no access to manifest-supplied command text,
paths, environment variables, or executable paths.

The host renders the returned quote using a small built-in SwiftUI quote card. It is
noninteractive and `.idleStatic`; its WixelsKit data source caches the first sample,
so palette-refresh ticks do not reroll it or launch another process. Existing dylib
plugins, themes, configuration, menus, package selection, and quarantine behavior
remain supported unchanged.

## Implementation Details

### 1. Script-widget manifest domain model and validation

**File**: `WixelsKit/Sources/WixelsKit/ScriptWidget.swift` (new)

**Current state**: WixelsKit exposes typed `WidgetSpec`, `Options`, placement, and
the erased widget interfaces, but has no data-only extension description.

**Modifications**:

- Define the public, versioned manifest model for the POC, with a schema version,
  stable lowercase-kebab-case `kind`, display label, default placement, fixed source
  identifier, and literal quote array.
- Limit supported source identifiers to `quote.random`; reject every other source.
- Validate an exact schema version, valid IDs, a nonempty quote list, bounded string
  count/length, and valid placement values before a manifest can be registered.
- Expose structured validation errors suitable for one concise host log entry per
  rejected manifest. Do not expose command, executable, path, or environment fields.
- Keep the type independent of TOMLKit and AppKit so it can be unit tested and used
  by both installed and app-resource packages.

### 2. Supervised Wixels-owned script broker and stable sample cache

**File**: `WixelsKit/Sources/WixelsKit/ScriptWidgetBroker.swift` (new)

**Current state**: `MusicMonitor` owns a specialized Perl child process and protocol
for private MediaRemote access. No reusable process wrapper or script-widget sample
cache exists. WixelsKit is a dynamic library consumed by the host, so the broker and
the widget that calls it must live here; WixelsKit cannot import a host type.

**Modifications**:

- Introduce a dedicated `actor` broker, following `MusicMonitor` rather than
  `@MainActor`, that resolves only the bundled
  `wixels-widget-broker.pl` resource, in the same resource root as
  `mediaremote-adapter.pl` (staging root in development, `Bundle.main` in release).
- Launch only `/usr/bin/perl` with that fixed resource path; set a minimal known
  environment, send one line-delimited JSON request, enforce a response-size limit
  and timeout, and terminate/reap failed children.
- Consume stdout asynchronously and await process termination through a continuation
  or task group raced against the timeout. Do not call `waitUntilExit()` or perform
  a synchronous pipe read on the broker actor; actor isolation alone does not make
  blocking work safe for a cooperative executor.
- Define typed request and response envelopes for `quote.random` rather than passing
  opaque dictionaries through the renderer.
- Treat malformed JSON, unknown response kinds, a nonzero exit, timeout, or empty
  quote as a recoverable unavailable state with one diagnostic, never a crash.
- Do not generalize this POC into a generic command runner or a long-lived listener.
  Listener/subscription lifecycle is a later phase once the static request protocol
  is proven.
- Follow `MusicMonitor`'s resource-root resolution directly inside WixelsKit. The
  broker must not depend on a path or type supplied by `Sources/wixels`.

**File**: `scripts/wixels-widget-broker.pl` (new)

**Modifications**:

- Implement the minimal newline-delimited JSON protocol: accept only the
  `quote.random` request with a literal string list and emit exactly one selected
  string envelope.
- Reject malformed input, unknown request types, empty arrays, overlong input, and
  non-string entries to stderr with a nonzero exit.
- Use no shell invocation, file reads, network calls, Apple Events, or environment-
  selected behavior. The script is Wixels-owned application code, not extension code.

### 3. Declarative script-widget discovery and registration

**File**: `Sources/wixels/ScriptWidgetLoader.swift` (new)

**Current state**: `PluginLoader` recursively finds only `libWidget*.dylib` and
`libTheme*.dylib`, then registers code through `wixels_register`. `PluginCatalog`
records groups and bare widget kinds for config/menu resolution.

**Modifications**:

- Discover `wixels-script-widget.json` only at the immediate package root under the
  app resource `script-widgets/` directory and
  `~/.config/wixels/script-widgets/<Package>/`.
- Decode and validate every manifest before registration; one malformed package must
  not prevent valid packages from loading.
- Convert a valid manifest to a WixelsKit-owned `WidgetSpec` whose build closure obtains
  `ScriptWidgetBroker` from shared WixelsKit `Services`; use the manifest's kind and placement
  so all existing TOML layout/menu behavior applies unchanged.
- Merge valid script-widget group/kind entries into `PluginCatalog` without changing
  dylib group selection or theme selection. Script widgets have no bundled theme and
  therefore use the existing global/fallback theme behavior only if they later become
  themed; the Quotes POC uses its own neutral renderer.
- Load dylib plugins first, then script-widget manifests. A script-widget kind that
  collides with an already-registered dylib kind logs and is skipped, so existing
  dylib widgets deterministically win. Script widgets never override installed code.
- Do not quarantine JSON-only manifests. They are declarative data that cannot name
  executable code; retain quarantine behavior for dylibs.

**File**: `Sources/wixels/main.swift`

**Modifications**:

- Load script widgets alongside existing plugin discovery before `buildSession()` so
  script kinds appear in the current menu/config resolver; do this after
  `PluginLoader.load(into:)` so dylib collision precedence is deterministic.
- Use the shared `Services.scriptWidgetBroker` supplied by WixelsKit in script-widget
  build closures, preserving the existing shared-service rule.
- Update the “no widgets installed” condition so valid script widgets count as
  installed widgets.

### 4. Native declarative Quotes renderer

**File**: `WixelsKit/Sources/WixelsKit/ScriptQuoteWidget.swift` (new)

**Current state**: `plugins/Cynaberii/Quotes` is an interactive themed Swift dylib
widget. It reads a configurable JSON file directly and rerolls on click; its SwiftUI
view is private to the plugin.

**Modifications**:

- Add a WixelsKit-owned `Wixel` implementation used only by a validated script-widget
  manifest. Its associated source owns a per-mount first-sample cache: the first
  `sample()` asks the broker, while every later sample returns that exact value.
  This prevents `.idleStatic` palette-refresh ticks from launching another child or
  rerolling the quote. A failed first sample likewise caches one deterministic
  unavailable message for the session.
- Render a simple, read-only native text card using existing palette primitives,
  bounded wrapping, and the manifest's stable placement. Do not copy Cynaberii's
  pixel-art view or add click-to-reroll in this POC.
- Use `.idleStatic` and `interactive = false`; later polling and actions will extend
  the data protocol only after this lifecycle works.
- Keep manifest data and samples `Sendable`/`Equatable` so the existing erasure and
  scheduler contracts remain intact.

### 5. Built-in POC package and resource staging

**File**: `script-widgets/Quotes/wixels-script-widget.json` (new)

**Modifications**:

- Add the shipped example package with kind `script-quotes`, a neutral default
  placement, source `quote.random`, and a small curated literal quote list.
- Keep data in the manifest rather than pointing at a user file; this demonstrates
  that third-party declarative packages cannot grant themselves filesystem access.

**File**: `build-plugins.sh`

**Modifications**:

- Stage `wixels-widget-broker.pl` beside `mediaremote-adapter.pl` and copy the
  built-in `script-widgets/Quotes/` resource tree verbatim for development runs.
- Mark only the copied broker script executable and keep the JSON manifest as data.
- Extend staging validation with a script-widget discovery check that does not depend
  on loading a dylib.
- Keep both source assets outside `Sources/`, matching the existing adapter source:
  SwiftPM must not process them or create a module resource bundle. Development
  resolution comes only from `WIXELS_PLUGIN_ROOT=$DEST`.

**File**: `package-app.sh`

**Modifications**:

- Copy the broker script and built-in script-widget resource tree verbatim into
  `Wixels.app/Contents/Resources`.
- Include the broker in architecture, minimum-OS, code-signing, archive round-trip,
  and resource-presence checks while continuing to keep dynamic extension packs out
  of the app bundle.

## Technical Considerations

- **Existing patterns**: Follow `MusicMonitor` for app-resource resolution and child
  cleanup, `PluginLoader`/`PluginCatalog` for discovery and grouping, `Services` for
  one shared source instance, `WidgetSpec`/`erase` for host mounting, and
  `ConfigTestSuite` for TOML/menu behavior.
- **Trust boundary**: The manifest is the extension API. It must be strict and
  closed-world: unsupported JSON keys should either be rejected or ignored according
  to an explicitly tested policy; no future “command” escape hatch is introduced.
- **Process lifecycle**: The POC uses a one-shot request to avoid orphaned listeners.
  The dedicated broker actor must close stdin/stdout, asynchronously observe child
  termination, and cap time/output without blocking an actor executor. Reuse the
  MediaRemote lifeline pattern only when a later `deflisten`-like capability is
  introduced.
- **Idle-static lifecycle**: `Scheduler.refreshOnce()` also ticks idle-static widgets
  when palettes change. The Quotes source must therefore cache its first result and
  never ask the broker again during this POC; quote content stays stable across a
  recolor and the stated one-child-process cost remains true.
- **Resource paths**: Development staging and packaged-app resources must resolve
  identically through the existing `WIXELS_PLUGIN_ROOT` staging root in development
  and `Bundle.main.resourceURL` in release. Keep source assets outside `Sources/`;
  do not use SwiftPM resources or rely on the current working directory.
- **Compatibility**: Existing dylib packs, TOML entries, themes, group layouts,
  quarantine prompts, and current `quotes` kinds remain untouched. The POC uses the
  distinct kind `script-quotes` to avoid collision.
- **Failure UX**: A script failure renders a stable unavailable state and logs once;
  it must not create a retry storm or prevent other widgets from mounting.

## Type Definitions

- `ScriptWidgetManifest`: decodable, validated data-only package definition.
- `ScriptWidgetSource`: closed enum initially containing only `quoteRandom`.
- `ScriptWidgetValidationError`: typed schema/field rejection reason.
- `ScriptWidgetRequest` / `ScriptWidgetResponse`: versioned line-delimited JSON
  envelopes for the fixed broker protocol.
- `ScriptWidgetBroker`: session-scoped WixelsKit service with typed quote sampling.
- `ScriptWidgetLoader`: host loader returning registered script specs and catalog
  membership; it does not load code.
- `ScriptQuoteWidget`: internal `Wixel` adapter from a validated manifest to the
  existing scheduler/render pipeline.

## Performance & Cost Impact

- One child process is launched only when the static Quotes widget takes its initial
  sample. Its cached result is reused for scheduler ticks caused by palette changes,
  so no periodic process spawning, polling, network access, or listener remains after
  that sample.
- The quote list and response are size-bounded to avoid memory abuse from a malformed
  manifest or helper output.
- Future shared broker capabilities should reuse a process only after a concrete
  streaming requirement is measured; this POC deliberately optimizes for lifecycle
  clarity and security over throughput.

## Backward Compatibility

- Dylib extension packages and their `wixels-package.json` schema remain unchanged.
- Existing `quotes` rows retain their current Cynaberii behavior. The POC registers
  `script-quotes`, so users opt in with a new TOML row.
- Missing script-widget directories are treated as an empty catalog. Wixels must run
  normally with no broker resource when no script widget is enabled; a missing broker
  only renders the script widget's unavailable state.
