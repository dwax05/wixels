# F 0.3.2 — Declarative Runtime Hygiene

## Overview

No new features. Fixes one real bug and pays down the incidental debt flagged in the
7c021b1...HEAD code review of the declarative runtime (F_0.3.0 / F_0.3.1 plus the
54b7b26 declarative-runtime commit). Architectural findings from that review
(TOMLKit config layer, `Wixel` seam exception) were resolved by documentation —
see CLAUDE.md "The declarative TOML widget runtime" — and are out of scope here.

## Problem Statement

- **Dead `on-click` (bug).** `DeclarativeWidget` hardcodes `let interactive = false`
  (`CommandTextWidget.swift:234`) while nodes attach `.onTapGesture` for `on-click`.
  `Host.swift:544` sets `w.ignoresMouseEvents = !m.interactive`, so every declarative
  widget is click-through and `on-click` can never fire. Interactive widgets also must
  sit at `.desktopIconWindow` level (CLAUDE.md sharp edge).
- **Missing spec'd diagnostics.** F_0.3.1 asked for two logs that never landed: a
  warning when a `listen` variable carries a meaningless `interval`
  (`WidgetsConfig.swift:118` ignores it silently), and a hint when `style` is a TOML
  table instead of a preset-name string (`resolveStyle`, `WidgetsConfig.swift:177`,
  yields nil with no log).
- **`CommandTextWidget.swift` is five modules in one file** with a name that matches
  none of them: `ProcessGroupRegistry`, `CommandVariableStore`, `DeclarativeWidget` +
  views, `WidgetsSession`, `interpolate`.
- **Local duplication and dead weight.** The TOML number coercion
  `t["k"]?.double ?? t["k"]?.int.map(Double.init)` is hand-rolled ~8 times though
  `WidgetStylePatch.parse` already has a `number()` helper; `DeclarativeNodeView.content`
  repeats the `visible.isVisible ? AnyView(decorated(container)) : AnyView(EmptyView())`
  shape verbatim for row/column/stack; `binding(_:visibility:context:)` is called with
  `visibility: true` at every site; `DeclarativeStyle` is a bare
  `{ let value: WidgetStyle }` wrapper; `WidgetsSession.mount` passes a magic
  `configIndex: -1` sentinel.
- **Listen backoff timer drift (minor).** `listenLoop` (`CommandTextWidget.swift:149`)
  measures the whole `runListen` call including `reap`'s SIGTERM → 2s → SIGKILL wait,
  so a crash-looping command that resists SIGTERM banks ~2s of fake "healthy" time
  per cycle toward the ≥30s backoff reset.

## Non-goals

- No behavior changes beyond the click fix and the two new log lines.
- Repeated `switch node` over `DeclarativeNode` (view / accessibility / test helper)
  stays: interpreter pattern, three switches over one enum is the idiomatic Swift
  shape; consolidating would fight value types for no reader benefit.
- `ValueBinding.source` string DSL stays a string at the boundary (it is the TOML
  surface); typed condition parsing is a future change if the DSL grows.

## Phases

### Phase 1 — Click fix (the only bug)

Derive `interactive` from the parsed tree: add a computed `containsAction` walk over
`DeclarativeNode` (a node or any descendant with a non-nil action) and set
`DeclarativeWidget.interactive` from it at init. Verify the mount path gives
interactive declarative widgets `.desktopIconWindow` level, same as compiled
interactive widgets; passive declarative widgets keep current click-through behavior.

- [x] `DeclarativeNode.containsAction`; `DeclarativeWidget.interactive` derived
- [x] Confirm window level + `ignoresMouseEvents` path in `Host.swift` for the mount
      (host derives everything from `m.interactive`: `InteractivePanel`,
      `ignoresMouseEvents = false`, base level `.desktopIconWindow + 1`)
- [x] Test: config with `on-click` mounts interactive; without stays passive
      (parse-level assertions in `runWidgetsConfigTestSuite`, 16/16 pass)
- [x] Live check: `on-click` fires (user-confirmed on child nodes and containers)
- [x] Scope addition (user-driven): `row`/`column`/`stack` now carry
      `action: DeclarativeAction?` — parsed from `on-click`, fired via `decorated`,
      counted by `containsAction`; whole-card clicks work (18/18 assertions)

### Phase 2 — Spec'd diagnostics

- [x] `WidgetsConfig` variable parsing: `kind = "listen"` + `interval` present →
      `Log.note` that interval is ignored for listen variables
- [x] `resolveStyle`: `row["style"]` is a table → `Log.note` hint to use a preset
      name plus top-level row keys
- [x] Widget-level `on-click` alongside `[widget.root]` → `Log.note` it is ignored,
      put it on a node (added after live-test confusion)
- [x] Extend `runWidgetsConfigTestSuite` for all three paths (21/21 pass)

### Phase 3 — File split (mechanical, no logic change)

Split `CommandTextWidget.swift` by module, names that say what they hold:

- `CommandRuntime.swift` — `ProcessGroupRegistry`, `CommandVariableStore`, spawn/reap,
  `CommandError`, `interpolate`
- `DeclarativeWidget.swift` — `DeclarativeWidget`, `DeclarativeWidgetView`
- `DeclarativeNodeView.swift` — `DeclarativeNodeView`, `DeclarativeImage` +
  cache/remote image views, `declarativeAccessibilityLabel`
- `WidgetsSession.swift` — `WidgetsSession`, `DeclarativeCommand`

- [x] Move types; keep `private` where possible, `internal` only where the split
      forces it (widened: `DeclarativeNodeView`, `declarativeAccessibilityLabel` —
      both consumed from `DeclarativeWidget.swift`)
- [x] `swift build` + full test-flag sweep green; no diff in behavior
      (layout suite needs `WIXELS_CONFIG` pointing at a temp desktop.toml with
      compiled widgets enabled — the live config is all-declarative and mounts 0)

### Phase 4 — Duplication & dead weight

- [x] Promote one shared TOML `number(_:)` helper (`TOMLTable.number` +
      `TOMLValueConvertible.asDouble` in `WidgetsConfig.swift`); all 8 hand-rolled
      sites replaced, present-but-invalid logging preserved
- [x] Extract the container render shape in `DeclarativeNodeView.content`
      (private `container(_:style:visible:action:)`; row/column/stack call it)
- [x] Drop `visibility` param from `binding(_:context:)` (all sites passed `true`)
- [x] Inline `DeclarativeStyle` → `WidgetStyle` directly (`Sendable` confirmed)
- [x] `configIndex: -1` → `WidgetHost.unmanagedConfigIndex` named constant
      (type change rejected: `Int?` ripples into layout persistence records)
- [x] `listenLoop`: `runListen` returns healthy duration measured before `reap`;
      threshold and backoff sequence unchanged

### Phase 5 — Verification

- [x] `swift build`; `--widgets-config-tests` (21/21), `--config-tests`,
      `--layout-tests` (temp `WIXELS_CONFIG`) all green
- [x] Startup proxy: `wixels up — 3 widget(s).` via `script -q` with live declarative
      config
- [x] Listen crash-loop check: `echo x; exit 1` listen variable restarts on 1s→2s→4s
      cadence, no backoff reset (healthy ≈ 0s per cycle). Known pre-existing noise:
      "recovered" logs on every received line, so a crash-looper that emits one line
      per cycle logs a recovered/exited pair each restart — semantics debatable,
      out of scope here.
